# 异步请求与会话切换问题 - 定位与优化方案

## 一、问题现象

1. **请求非异步**：AI 回复期间，前端等待整次请求完成后才更新
2. **无法切换会话**：AI 正在回复时，无法切换到其他会话

---

## 二、问题定位

### 2.1 前端：会话切换被 `isProcessing` 阻塞

**文件**：`Assistant/frontend/js/app.js`

| 位置 | 代码 | 说明 |
|------|------|------|
| 251-252, 258 | `if (conversation.id === activeConversationId \|\| isProcessing) return;` | 点击会话项时，若 `isProcessing` 为 true 则直接返回，禁止切换 |
| 285-286 | `if (!conversationId \|\| isProcessing) return;` | 删除会话对话框被阻止 |
| 302 | `if (!conversationId \|\| isProcessing) return;` | 内联编辑标题被阻止 |
| 447 | `if (isProcessing) return;` | 新建会话被阻止 |
| 414, 434 | `isProcessing = true` → `finally { isProcessing = false }` | 整个请求期间 `isProcessing` 为 true |

**结论**：`isProcessing` 在 AI 请求全流程中置为 true，导致切换、新建、编辑、删除会话全部不可用。

---

### 2.2 前端：单次请求等待完整响应（非流式）

**文件**：`Assistant/frontend/js/app.js`，约 426-434 行

```javascript
const data = await apiFetch(`/conversations/${conversationId}/messages`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ content: trimmed }),
});
```

- 使用 `await apiFetch(...)` 一次性等待完整 JSON 返回
- 无 SSE / WebSocket 等流式接口
- 后端需要完成：LLM 生成 + TTS 合成，耗时可能达数十秒

---

### 2.3 后端：同步阻塞调用

**文件**：`Assistant/backend/main.py`，约 108-159 行

```python
@app.post("/conversations/{conversation_id}/messages")
async def create_conversation_message_endpoint(...):
    ...
    reply = chat(model_messages)  # 同步阻塞！
    ...
    tts_url, _ = synthesize_to_file(...)  # 可能也是同步
```

**文件**：`Assistant/backend/services/qwen_chat.py`，约 83 行

```python
resp = requests.post(url, headers=headers, json=payload, timeout=120)
```

- `chat()` 使用 `requests.post`，是同步 I/O
- 在 `async def` 中直接调用，会阻塞 FastAPI 事件循环
- 一个长请求期间，其他请求（如切换会话、拉取消息列表）也会被拖慢

---

## 三、优化步骤规划

### 步骤 1：前端允许在 AI 回复期间切换会话（快速修复）

**目标**：AI 回复时仍可切换会话、新建会话、编辑标题、删除会话。

**改动**：

1. **移除或调整 `isProcessing` 对会话切换的阻挡**  
   - 点击会话项时：去掉 `isProcessing` 限制，或在 `isProcessing` 时仍允许切换，但给出提示
2. **区分「当前请求所属会话」与「当前展示会话」**  
   - 用 `requestingConversationId` 记录正在请求的会话  
   - 用 `activeConversationId` 记录当前展示的会话  
   - 切换会话时只影响 `activeConversationId`，不影响正在进行的请求
3. **后台请求完成后正确写回数据**  
   - 若完成时 `requestingConversationId === activeConversationId`：正常更新当前展示  
   - 若用户已切换到其他会话：更新对应会话数据，不强制切回

**预期**：用户可随时切换会话，AI 回复在后台继续，不影响其他操作。

---

### 步骤 2：前端将请求与会话解耦（架构优化）

**目标**：请求状态与会话切换解耦，避免全局 `isProcessing` 影响所有功能。

**改动**：

1. **按会话记录请求状态**  
   - 使用 `Map<conversationId, { status, abortController }>`  
   - 每个会话有独立的 `loading` / `error` 状态
2. **仅禁用「当前展示会话」的发送**  
   - 若当前会话正在请求，则禁用输入框和发送按钮  
   - 其它会话仍可正常点击、查看、编辑
3. **支持 AbortController 取消**  
   - 切换会话或关闭页面时，可取消未完成请求（若后端支持）

---

### 步骤 3：后端改为非阻塞（asyncio 友好）

**目标**：不阻塞事件循环，支持多请求并发。

**改动**：

1. **使用异步 HTTP 客户端**  
   - 将 `requests.post` 替换为 `httpx.AsyncClient` 或 `aiohttp`  
   - 例如：`async with httpx.AsyncClient() as client: await client.post(...)`
2. **在 FastAPI 中正确调用**  
   - `chat()` 改为 `async def chat(...)`  
   - 在 `create_conversation_message_endpoint` 中 `await chat(...)`
3. **TTS 等耗时操作异步化**  
   - 若 `synthesize_to_file` 是同步的，可用 `asyncio.to_thread()` 放到线程池  
   - 或改为 `async def synthesize_to_file(...)` 使用异步 I/O

---

### 步骤 4（可选）：流式输出 + 先文本后 TTS

**目标**：更快看到首字，并允许在生成过程中切换会话。

**改动**：

1. **后端 SSE 流式接口**  
   - 新增 `/conversations/{id}/messages/stream`  
   - 使用 `StreamingResponse`，按 token 或 chunk 输出
2. **前端改用 EventSource / fetch + ReadableStream**  
   - 逐步追加到 UI，而不是等完整 JSON
3. **TTS 异步生成**  
   - 文本流结束后再触发 TTS  
   - 或使用 WebSocket 推送 TTS 完成后的 URL

---

## 四、推荐实施顺序

| 优先级 | 步骤 | 工作量 | 收益 |
|--------|------|--------|------|
| P0 | 步骤 1：允许 AI 回复期间切换会话 | 小 | 立即解决「不能切换」问题 |
| P1 | 步骤 2：请求与会话解耦 | 中 | 交互更清晰，状态更易维护 |
| P2 | 步骤 3：后端异步化 | 中 | 提升并发和稳定性 |
| P3 | 步骤 4：流式输出 | 大 | 首字延迟更低，体验更好 |
